Models Tested

G

Gemma3:4b

Google • 4 Billion Parameters

A lightweight model from Google's Gemma family designed for efficient local deployment. Strong at combinatorics and limit evaluation.

P

Phi3:3.8b

Microsoft • 3.8 Billion Parameters

Microsoft's compact Phi-3 model optimized for fast inference. The smallest model tested, with the fastest response times but lower accuracy.

Q

Qwen3:4b

Alibaba • 4 Billion Parameters

Alibaba's Qwen3 model balancing capability and size. Achieves the highest overall accuracy but requires significantly more inference time.

Why these models? All three are free, open-source, and small enough to run on consumer hardware using Ollama for local inference. This keeps the evaluation cost-free and fully reproducible without requiring cloud API access.

Extraction Strategies

The system uses multiple strategies to extract the final answer from each model's response, applied in priority order:

  1. 1
    FINAL_ANSWER keyword — Looks for explicit "FINAL_ANSWER:" markers in the response
  2. 2
    Boxed notation — Matches LaTeX \boxed{} patterns commonly used by math-trained models
  3. 3
    Keyword search — Finds phrases like "the answer is" or "therefore"
  4. 4
    Last value fallback — Extracts the last numerical or expression value from the response

Comparison Methods

Once answers are extracted and normalized, they are compared using type-aware matching:

Exact match

String-identical after normalization

Equivalent match

Mathematically equal expressions (e.g., 2x+4 vs 4+2x)

Tolerance match

Numerically close within a small epsilon (for rounding)

No match

Extracted answer does not match expected answer

Type mismatch

Answer types differ (e.g., fraction vs decimal) making comparison unreliable

Metrics Collected

Accuracy % Correct answers / total questions

Match Type Exact, equivalent, tolerance, no_match, etc.

Fairness & Reproducibility

Identical prompts — every model receives the exact same question text
Same hardware — all tests run on the same machine via Ollama
Consistent configuration — same Ollama settings for all runs
Automated verification — no manual scoring or subjective grading
Open source — full pipeline code and results are available for review
Repeatable — pipeline supports multiple runs for consistency verification
